# Teamregeln

## Rollen

| Rolle | Person | Aufgaben |
|-------|--------|----------|
| Mentor | Mario | System strukturieren, PRs reviewen, Regeln pflegen, Missionen definieren |
| Schüler | Elvis | Missionen ausführen, Ideen einreichen, PRs erstellen, Reviews lernen |

Beide Rollen sind gleichwertig — der Mentor lernt mit, der Schüler gestaltet mit.

## Täglicher Rhythmus

1. Repo öffnen — was ist neu?
2. Offene PRs checken — gibt es was zu reviewen?
3. Aktive Mission anschauen — was ist der nächste Schritt?
4. Arbeiten, committen, PR erstellen
5. Auf Review warten oder Review geben

## Branch-Regeln

- `main` bleibt immer sauber und funktionsfähig
- Pro Aufgabe ein eigener Branch
- Branch-Name folgt dem Schema aus CONTRIBUTING.md
- Branch wird nach Merge gelöscht

## PR-Regeln

- Jede Änderung braucht einen Pull Request
- Jeder PR braucht ein Review
- PR-Template immer ausfüllen
- Kleine PRs > große PRs
- Kein Merge ohne Approve

## Kommunikation

- PR-Beschreibung erklärt: **Was**, **Warum**, **Wie testen**
- Review-Kommentare sind sachlich und konkret
- Bei Unklarheiten: lieber nachfragen als raten
- GitHub-Notifications aktiv halten (auch auf dem Handy)

## Konflikte

- Merge-Konflikte sind kein Fehler — sie sind eine Entscheidung
- Konflikte werden bewusst und gemeinsam gelöst
- Die inhaltlich richtige Version gewinnt, nicht die zuletzt gepushte

## Definition of Done

Eine Aufgabe ist fertig, wenn:

1. Sie in einem passenden Branch bearbeitet wurde
2. Die Änderung nachvollziehbar committet wurde
3. Der Branch gepusht wurde
4. Ein Pull Request erstellt wurde
5. Die andere Person reviewed hat
6. Korrekturen eingepflegt wurden
7. Der PR gemerged wurde
8. Der Branch gelöscht wurde
9. Die Änderung sichtbar ist (App, Daten oder Docs)
